## Project Scale & Constraints

`E` **Zero manually-written code constraint.** Team of 3→7 engineers built internal product entirely via Codex over 5 months from empty repo. Humans steer; agents execute all code including tests, CI, docs, tooling, observability, and internal utilities.

`E` ~1M LOC, ~1500 PRs merged, 3.5 PRs/engineer/day avg. Throughput *increased* as team grew. Single Codex runs regularly work 6+ hours. Initial scaffold (repo structure, CI, formatting, framework, `AGENTS.md`) generated by Codex CLI + GPT-5.

---

## Engineer Role Redefined

`L` **Engineering becomes environment design.** Primary job: build tools, abstractions, and feedback loops enabling agents. When agent fails → diagnose missing capability, not "try harder."

`P` **Prompt-driven workflow.** Engineer describes task → agent opens PR → agent self-reviews + requests agent reviews → iterates in feedback loop until all agent reviewers satisfied (Ralph Wiggum Loop). Human review optional; shifted mostly agent-to-agent over time.

---

## Application Legibility

`L` **Bottleneck shifted from coding to QA** as code throughput grew → solution: make app itself legible to agents.

`P` **App bootable per git worktree** — Codex launches isolated instance per change. Chrome DevTools Protocol wired into agent runtime via skills: DOM snapshots, screenshots, navigation → agent reproduces bugs, validates fixes, reasons about UI directly.

`P` **Ephemeral local observability stack per worktree** — logs (LogQL), metrics (PromQL), traces (TraceQL) exposed to agent. Enables prompts like "ensure startup < 800ms" or "no span > 2s in critical journeys." Stack torn down after task completes.

## Repository Knowledge as System of Record

`L` **What the agent can't access in-context doesn't exist.** Slack, Google Docs, tacit knowledge → invisible. All context must be repo-local, versioned artifacts (code, markdown, schemas, exec plans).

`L` **Map, not manual.** Large monolithic `AGENTS.md` failed — crowds out task context, becomes non-guidance when everything's "important", rots instantly, resists mechanical verification.

`P` **Progressive disclosure.** Short ~100-line `AGENTS.md` as table of contents → pointers into structured `docs/` tree: `design-docs/`, `exec-plans/{active,completed}`, `product-specs/`, `references/`, plus `ARCHITECTURE.md` and domain guides (`FRONTEND.md`, `SECURITY.md`, etc.). Quality document grades each domain/layer, tracking gaps.

`P` **Mechanical freshness enforcement.** Linters + CI validate knowledge base structure, cross-linking, currency. Recurring "doc-gardening" agent scans for stale docs → opens fix-up PRs.

`P` **Plans as first-class versioned artifacts.** Lightweight plans for small changes; execution plans with progress + decision logs for complex work. Active, completed, and tech-debt plans co-located in repo.

---

## Agent Legibility & Dependencies

`L` **Optimize for agent legibility, not human style.** Code that is correct, maintainable, and legible to future agent runs meets the bar — human stylistic preferences secondary.

`L` **Favor "boring" composable tech** — stable APIs, well-represented in training data → easier for agents to model. Sometimes cheaper to reimplement than work around opaque upstream behavior.

> `E` Reimplemented `p-limit`-style concurrency helper instead of pulling package — tightly integrated with OTel, 100% test coverage, exact runtime semantics.

---

## Enforcing Architecture & Taste

`L` **Enforce invariants mechanically, not implementations.** Agents thrive with strict boundaries + predictable structure + local autonomy within those boundaries.

`P` **Layered domain architecture.** Each business domain → fixed layers: Types → Config → Repo → Service → Runtime → UI. Cross-cutting concerns (auth, telemetry, feature flags) enter via single Providers interface. Dependency directions strictly validated via custom linters + structural tests.

`P` **Custom lints with remediation messages.** Enforce structured logging, naming conventions, file size limits, platform requirements. Error messages contain fix instructions → injected into agent context on failure.

`L` **Promote docs into code.** When documentation falls short at preventing drift, encode the rule as a lint or test. Human taste captured once → enforced continuously.

> `E` Require parsing at boundaries ("parse, don't validate"). Agent converged on Zod without being told — not prescriptive on specific library.

---

## Merge Philosophy

`L` **High throughput changes merge norms.** Minimal blocking gates, short-lived PRs, test flakes addressed via follow-up runs rather than blocking. Corrections cheap; waiting expensive.

`C` "This would be irresponsible in a low-throughput environment." — applicability limited to systems where agent throughput far exceeds human attention.

---

## Full Agent Autonomy

`E` **End-to-end feature loop from single prompt:** validate codebase → reproduce bug → record failure video → implement fix → validate fix → record resolution video → open PR → respond to feedback → detect/remediate build failures → escalate only when judgment needed → merge.

`C` "Depends heavily on the specific structure and tooling of this repository and should not be assumed to generalize without similar investment — at least, not yet."

---

## Entropy & Garbage Collection

`L` **Agents replicate existing patterns — including bad ones** → drift inevitable. Manual cleanup ("AI slop Fridays" — 20% of week) didn't scale.

`P` **Golden principles + automated cleanup.** Opinionated mechanical rules encoded in repo. Background Codex tasks on regular cadence: scan deviations, update quality grades, open targeted refactoring PRs (most reviewable in <1 min, auto-merged).

> `E` Golden principles include: prefer shared utility packages over hand-rolled helpers; validate at boundaries or use typed SDKs, never probe data YOLO-style.

`L` **Tech debt = high-interest loan** — continuous small increments >> compound painful bursts.
